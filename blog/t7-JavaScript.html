Snakes, Stones & Coffee: Comparing Python, Ruby and JavaScript

January 18, 2015

This week in Dev Bootcamp, we switched gears from Ruby to JavaScript in our challenges. I had hacked my way through some JavaScript before, but had always felt somewhat intimidated by it for some reason. I think that coming from Python, JavaScript syntax felt a lot messier and more difficult to understand at the time, though learning Ruby before diving into JavaScript this time around felt much more natural. Just the little bit of extra training that Ruby provided with things like remembering to close off blocks of code made it much easier to come from Ruby to JavaScript, compared to from Python.

After a week of focusing solely on JavaScript, one of the most significant differences between it and Ruby is how the languages create new objects. Ruby is a lot like Python, in that it relies on using classes as a kind of blueprint for making new objects. Here's an example of how you'd create an object in Ruby: 

          		
class Car
  def initialize(make, model, year, color)
    @make = make
    @model = model
    @year = year
    @color = color
    @speed = 0
  end

  def changeSpeed(mph)
    @speed = mph
  end
end

car = Car.new(“Ford”, “Mustang”, 1969, “red”)

And now, here's how you'd create the same object in JavaScript: 

          		
var car = {
  make: “Ford”,
  model: “Mustang”,
  year: 1969,
  color: “red”,
  speed: 0,
  changeSpeed: function(mph) {
    this.speed = mph;
  }
}

Right off the bat, there's a big difference- no mention of class. Although JavaScript is also an object-oriented programming language, it doesn't actually have any classes. Once you look past the lack of a class keyword, you can start to see some similarities between the two languages when initializing an object. The newly minted objects from both JavaScript and Ruby have the same properties, and it's fairly easy to see that this.speed is similar to Ruby's @speed, in that it is JavaScript's way of accessing the object's speed property from within the object.

Since there are no classes in JavaScript, how do you write DRY code in JavaScript? If you had to create 100 Car objects, you can see how bloated your software would get, and how much of a pain it would be if you wanted to add the license plate number as a property for each of those objects. JavaScript handles this through object prototypes. If classes are blueprints which instruct how a house should be built, then prototypes would be a model home in a sub-development. The blueprint isn't a house itself, but it's used to let the builders know how many rooms it should have, what the square footage should be ect... In contrast, a model home (the prototype) is a fully functioning house in it's own right, but, like the blueprint, is used to define what characteristics other houses like it have.
